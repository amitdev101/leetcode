Let's say we have a web application which is used by 20 users daily. One day 10,000 users want to use your app at the same time. But your server can't handle that much traffic and starts responding with errors. The problem here is not in the hardware or network. It is caused by how software handles connections. But what exactly is a connection? When a browser wants something from the server, it first opens a channel. That open channel is the connection. The browser sends requests through it like for a JavaScript file and the server uses that same channel to send the data back. One connection can carry many requests one after another. So it isn't the same thing as a single request and it stays open the whole time even when nothing is being sent. In the late 90s and early 2000s, web servers used a thread per connection model. So when a user connects, the server spawns a new thread or process to handle them. that thread stays alive as long as the connection remains open. If you have 100 visitors, you have a 100 threads. If you have 10,000 visitors, you need 10,000 threads. Every thread requires its own stack memory. Even if you shrink the stack size aggressively, a thread might still take a megabyte of RAM. 10,000 threads mean 10 GB of memory consumed just to keep the connections open before the application even does any work. Another problem comes down to CPU limitations. A processor can only execute one thing at a time per core. To fake parallel execution, the operating system rapidly switches between threads, saving the state of one thread and loading the state of another takes time. At 10,000 threads, the CPU spends most of its cycles just switching contexts instead of serving web pages. This is a well-known problem with its own name, the C10K problem. The C is for concurrent, the 10K for 10,000 connections. A Russian engineer hit exactly this problem at Rambler. the search portal he worked for and he created engine X to fix it. This thread per connection model is wasteful because a connection spends most of its life waiting. It waits for the client to send a request, waits for the disk or a backend to answer, waits for the network to carry the bytes back. The whole time the thread tied to that connection is blocked. It can't pick up anyone else. It just holds its memory until that one connection finally needs something. Engine X uses a single process to watch thousands of connections at the same time. At any moment, most of them are idle. So, it only does anything when one of them has data coming in or a response going out. It handles that connection, then moves to the next one that needs something. The process never stops to wait on a connection the way the old thread did. So, one process keeps thousands of them going. When you start engine X, it creates one master process and multiple worker processes. The master process does very little actual work. It reads the configuration file, binds to the network ports, and spawns the workers. The workers do the actual job of handling network traffic. You usually configure Engine X to run one worker process per CPU core. If you have an 8 core server, you run eight workers. This specific ratio is intentional. When we have one worker per core, the operating system rarely needs a context switch between them. The worker process stays on the CPU and runs constantly to process network requests without interruption. We have one worker responsible for thousands of connections, but how does it know which ones actually need attention? most of them are doing absolutely nothing. So it can't waste time walking down the whole list and checking each connection one by one whether it has data. So instead of checking each connection itself, the worker lets the kernel do the watching. The kernel receives all the network traffic anyway. So it already knows which connections have data and which ones are idle. The worker uses a Linux facility called eole which keeps two lists that the worker and the kernel use to talk to each other. The first one is the interest list where the worker writes down which connections it wants the colonel to watch. The second one is the ready list where the kernel writes back which of those connections actually have something happening. The interest list gets built up as connections arrive. When a connection opens, the worker registers it once with a call named epolect and that adds it to the interest list. The kernel needs this list to know which connections belong to this worker. Meanwhile, the ready list is filled by the kernel. When data arrives for a connection that is on the interest list, the colonel adds that connection to the ready list. Let's look at a real example. A user opens the web app and that opens a connection. The worker sees the new connection and uses eole to tell the kernel to watch it. So the connection lands on the interest list. Then the user clicks the orders tab and the front end sends a request to the server to fetch those orders. That request travels over the connection that's already open. When the kernel spots the incoming bite, it checks the interest list and immediately flags the connection on the ready list. Once the connection is on the ready list, the colonel's job is done. The worker still has to read the bytes, fetch the orders, and send the response. To actually process the work, the worker runs an endless loop. The first thing it does is call eul wait. This asks the kernel for connections that are sitting on the ready list. If our connection is there, the worker grabs it, handles the request, and ships the data back to the user. Then the whole cycle starts all over again. If the ready list is empty, e-le blocks the endless loop. The worker falls asleep on that exact line of code without draining any processor resources. The loop doesn't just run over and over asking the kernel if anything changed. It asks once and waits for the kernel to wake it up when a new request arrives. This endless cycle runs for the entire life of the process. And this is called the event loop. Since the worker never wastes time on idle clients, Engine X can handle 10,000 connections at once. The event loop seems like a perfect solution. But it has one big problem. If the worker gets stuck on one connection, all other connections have to wait. Let's suppose your server stores large video files on a slow mechanical hard drive. One day a user requests one of those videos. So the connection arrives and the worker asks the drive for the data. Because the disc is slow, the worker sits there doing absolutely nothing, just waiting for the hardware to read the entire file and respond. That same worker is responsible for thousands of other active connections. While it waits for the video, everyone else is blocked until the disc responds. Someone else might be asking for a simple JavaScript file, but they won't get it until the slow disk read finally finishes. The system needed a way to handle slow operations without forcing everyone to wait. So in order to deal with those cases, developers built thread pools. When the worker receives a request that requires a slow disk read, it refuses to process it directly. It pushes the task to a queue and a separate background thread picks it up. The background thread takes over the job of talking to the slow disk, leaving the main worker free to return to the event loop. It goes right back to calling eole weight and serving the other users. Once the background thread finally pulls the file from the disk, it puts the data into memory and sends a signal. The next time the main worker checks the ready list, it sees that the data is ready. It grabs the bytes, ships them to the user, and immediately moves on to the next task. The main process never had to stop and wait for a slow operation that would block all other connections. I said earlier that the worker grabs the connection, handles the request, and ships the data back. But I think we need to define what handling the request actually means. So the last thing I want to cover in this video is what Engine X is actually responsible for. Engine X does not run your application code. It acts as a reverse proxy. When a request arrives, it opens a separate connection to your backend like a node API. It forwards the incoming data to node, waits for the backend to finish its work, and relays the response back to the user. Because engine X already sits in the middle and intercepts all the traffic, you can also use it as a load balancer. You just run multiple instances of your application and it will distribute the request across all your servers. So, the old approach of giving every single connection its own thread is what kills web servers the moment traffic spikes. Engine X completely changes this dynamic by relying on an event loop, allowing a single worker process to efficiently manage thousands of clients at the exact same time. Instead of wasting massive amounts of memory and processing power on idle connections that are doing absolutely nothing, it only uses server resources when there is actual data ready to be processed.
