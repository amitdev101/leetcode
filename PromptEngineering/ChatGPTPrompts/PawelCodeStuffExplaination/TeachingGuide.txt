1. Docker-under-the-hood.txt
I'm guessing you've used Docker at least once, but have you ever wondered how it works under the hood? You write a Docker file, you build an image, and you run a container. Your code is now running completely cut off from the rest of your machine. It has its own file system, its own network interfaces, and its own process tree. The code behaves exactly the same on your laptop as it does on a production server. And usually, that's enough. As long as the container spins up, we don't really care about what's under the hood. But in this video, we're going to see exactly what actually happens behind that docker run command. We often compare containers to virtual machines. A virtual machine uses a hypervisor to create an entirely simulated computer. It boots a kernel, loads hardware drivers, and runs a complete operating system from top to bottom. If you run three virtual machines on a single physical server, you have three separate kernels loaded in memory. That guarantees strong isolation, but it consumes a massive amount of memory and takes a long time to boot. A container is different. It never boots an operating system and it doesn't need a kernel. It is just a normal Linux process. It runs directly on the host and shares the kernel with every other container on the machine. Since you skip the entire OS boot sequence, the container starts almost instantly and takes up only the memory your application actually uses. If a container is just a normal process, we face a problem. Usually processes on a Linux machine can see each other. They share the same file system, the same network interfaces, and the same host name. To make a single process behave like a self-contained environment, the kernel has to completely block its view of the rest of the host. It does this using a Linux feature called namespaces. Namespaces control what a process can see. Let's start with the PID namespace, which handles process IDs. Normally, your operating system keeps one giant list of every running program. If you start a web server, the kernel assigns it a unique number like process 4500. Because it lives on that global list, your web server can technically look around and see every other application running on your machine. When you attach a P namespace to that process, the kernel puts a wall around it. Now, when the web server asks the operating system what else is running, the kernel filters the response. The process sees an empty system and assumes it must be the very first program that booted up. It labels itself as process ID 1. But if you open a terminal on your host machine, you will still see it sitting right there as process 4500. It is the exact same process just experiencing a different reality. There are several other namespaces working together. The mount namespace gives the process its own isolated view of the file system. The network namespace gives it a private network stack. The UTS namespace lets the process have its own host name. There are also namespaces for interprocess communication and user ids. You can map the root user inside the container to an unprivileged user on the host. By combining these namespaces, the process is completely isolated from the rest of the host. It genuinely believes it has the entire computer to itself. Namespaces control visibility, but they do nothing about resources. A process isolated by namespaces can still consume all the host CPU and crash the machine. The Linux kernel solves this with control groups, usually called Croups. While namespaces restrict what a process can see, croups restrict what a process can use. When you pass the memory or CPU flags to your Docker command, you are configuring these croups. Under the hood, the kernel manages them through a virtual file system. It creates a specific directory structure. You just write a number into a file and that sets the exact resource limit for a process. From that moment on, the kernel tracks exactly how much RAM and CPU time the container consumes. When the container reaches for more memory than its limit allows, the kernel tries to free up room first by reclaiming memory the container is already holding. If that's not enough, it kills the process. This guarantees one runaway container cannot starve the rest of the server. We mentioned earlier that the mount namespace gives a container its own view of the file system. But to actually start the process needs a root directory. We cannot let it use the hosts. So we have to provide a completely separate one. Long before Docker existed, Unix systems handled this with a command called chroot. You point the process at a specific folder and the kernel treats that folder as the absolute top of the file tree. So the process can't see anything outside that folder. But there is one major problem with chroot. It was never meant to be a security boundary. A process running as root can break out of it and get back to the host files. Modern container engines require something stronger. So they use a system call named pivot root. Instead of just restricting the view, pivot root completely replaces the root directory. The container gets a new isolated file system and the host machine's original file tree is safely disconnected. But pivot root is just a mechanism for the swap. It secures a boundary, but that new root directory cannot be empty. It has to be pre-filled with system libraries and your application. Providing that fully packed environment is exactly what a Docker image does. But an image is not a single large file. It is a stack of independent layers combined using a union file system, most commonly overlay FS. When you write a Docker file, each instruction that changes the file system creates a layer. The base operating system is a layer. Installing a dependency adds another layer. Copying your source code adds one more. Instructions that only set metadata, like envile system. They just attach settings to the image. Overlay FS merges all these directories together presenting them to the container as one unified file system. These image layers are strictly read only. When you start a container, Docker adds a thin writable layer on top. If the container modifies an existing file, it never touches the original. Overlay FS copies that file up into the writable layer and changes the copy. The container thinks it edited the file, but in reality, it just stacked a new version on top and hid the old one underneath. This design saves massive amounts of disk space. If you run 10 containers from the same image, they all share the exact same readonly layers in memory and on disk. This also explains why the order of instructions in your Docker file matters. If you change a line near the top, Docker must rebuild that layer and every subsequent layer below it. We still need to explain how the container talks to the outside world. The network name space left the container completely disconnected. It has its own network stack, but no physical network interface. Docker solves this by creating a virtual Ethernet pair. You can think of it as a virtual network cable. One end of the cable plugs into the container network namespace. The other end plugs into a virtual switch on the host machine called a bridge network, which by default is named Docker zero. When the container makes a request, the traffic travels through the virtual cable to the bridge. The host machine then routes that traffic to the physical network adapter. If two containers are connected to the same bridge, they can talk to each other directly using internal IP addresses. To expose a container to the public internet, you need to map ports. When you bind port 8080 on the host to port 80 in the container, Docker configures the host operating system to intercept traffic arriving at port 8080. The host uses network address translation to forward those packets through the bridge across that virtual cable and into the container. We have covered the core Linux features. We can now look at the actual architecture triggered by the docker run command. When you type the command, the docker command line interface does not start the container. It makes an API call to a background service called the docker demon. The demon manages images, networks, and storage volumes. It then passes the container specification to a component named containerd. Container D manages the life cycle of the container. It handles downloading the image and unpacking the layers. Even containerd does not create the process itself. It hands the final configuration to a low-level tool called run C. The only job run C has is to talk to the Linux kernel. It sets up the name spaces, configures the croups, unpacks the root file system, and starts the process. Then the moment that process is running, run immediately terminates. It builds the container and simply walks away. But on Linux, every process needs a parent. Since run is gone, a small process called the shim takes over and adopts the container. It stays for the rest of the container's life. It keeps the process alive even if containerd restarts and it grabs the exit code when the container finally stops. Let's summarize everything. A container is nothing but a regular Linux process. Namespaces isolate what it can see. Croups restrict what it can consume. Overlay FS handles layers to build its file system. And a virtual bridge wires it to the host network. I get asked all the time about how I build my animations. So, if you want to learn how to do it yourself, check the link in the description. 2. Redis-teaching.txt
Most developers start using Redis when their database gets slow and the API starts showing higher latency. In that case, someone will probably tell you to use cache. So, you set up Redis, store some data in it, and suddenly latency drops and everything is fine. For a lot of us, the mental model of Redis stops right there. We think of it as a temporary box for key-value pairs, a simple hash map running in memory that loses all its data the moment the server reboots. But if Redis is just a temporary dumb cache, why do modern tech stacks use it for a lot of stuff like, for example, message brokering, real-time analytics, geospatial tracking, or even as a primary persistent database? The truth is, saying that Redis is only for caching is like saying a phone is only for ringing. It's technically true, but it misses a lot of things. So, in this video, we will look at what's happening under the hood of Redis to understand how it actually works, why it's single-threaded, and how it manages to be both insanely fast and persistent at the same time. Let's start with the most obvious question. Why is Redis so much faster than a traditional database? If you query a relational database, the engine has to do a lot of work. It parses the SQL query, plans the execution, traverses a B-tree index, and eventually reads the data. But the biggest bottleneck is that data lives in the disk. Even with modern NVMe SSDs, disk access is measured in microseconds. As you probably know, Redis takes a different approach. It is in-memory data structure store, so it bypasses the disk for read and write operations. The data lives in RAM, and reading from it takes nanoseconds. The operations done on RAM are a lot faster compared to disk operations. But raw speed isn't enough. If you just wanted to store data in RAM, you could declare a global variable in your Node or Python back end, and it would do the same job. What makes Redis powerful is the second part of its definition, data structure store. Traditional caches only understand strings. You give them a key, and they store a massive blob of text. If you want to update one field in cached JSON object, you have to retrieve the entire string, then parse it, then update the data, then stringify it, and finally, you can save the updated JSON. One of Redis's advantages is that it natively understands various data structures. Out of the box, it supports lists, sets, hashes, and sorted sets. Each one comes with operations built specifically for it, so you can change the data in place without pulling the whole thing back to your app. Let's say you're building a real-time gaming leaderboard. In a traditional database, you'd have to sort all the rows by score from highest to lowest on every single request. Doing that on millions of rows for every page refresh is just really slow. In Redis, you use a sorted set. Under the hood, a sorted set is implemented using a hash table and a skip list. When you insert a player's score, Redis does the sorting right then in memory. When you need the top 10 players, there is no sorting left to do. You aren't just storing data. You are shifting the computation from your application layer directly into the memory layer. Now, this brings us to one of the most counterintuitive aspects of Redis. How does it handle thousands of concurrent client connections? If you have 10,000 users hitting your API, and that API is hitting Redis, you might assume Redis spins up thousands of threads to handle them, just like a traditional web server would, but it doesn't. It's single-threaded. When people hear this, they assume it must be slow. How can a single thread process 100,000 requests per second? To understand this, we have to look at what actually makes a multi-threaded application slow. When multiple threads try to access the same piece of memory, you have to use locks or mutexes to prevent data corruption. Thread A locks the data, thread B waits. The CPU wastes valuable cycles just switching between threads and managing who has access to what. Redis avoids this entirely because all the data is in RAM, reading or writing a value takes almost zero time. The real bottleneck is the network, waiting for the data packets to travel from the client to the server. So, Redis uses a technique called IO multiplexing. Specifically, it uses system calls like epoll on Linux. Instead of assigning one thread to wait for one network connection, a single thread monitors thousands of connections simultaneously. When a network socket has a command ready to execute, Redis takes that command, puts it in a queue, and executes it. Because it's single-threaded, commands are executed strictly one after the other. This guarantees atomic operations. You never have to worry about race conditions. If two users try to decrement an inventory counter at the exact same millisecond, Redis guarantees one will execute fully before the other begins. There is no need for locks. One thing worth mentioning is that modern versions of Redis use additional threads, but only for parsing network stuff. The actual execution of commands, the part that touches the data, remains strictly single-threaded to preserve this lock-free speed. So, we have an incredibly fast single-threaded execution loop running entirely in RAM. But, RAM is volatile. If the server loses power, the RAM is cleared, so it means that your data is gone. If you are using Redis purely as a temporary cache, maybe you don't care. But, if you are using it to store user sessions or shopping carts, data loss can be a problem. This is where the hidden complexity of Redis persistence comes in. Redis solves this problem using two completely different mechanisms, RDB and AOF. RDB stands for Redis database. It takes point-in-time snapshots of your entire data set and saves them to disk. But, if Redis is single-threaded and writing 10 GB of data to an SSD takes several seconds, wouldn't taking a snapshot freeze the entire database and block all incoming requests? It would if Redis didn't use a brilliant operating system trick. When it's time to take a snapshot, Redis asks the Linux kernel to execute a fork system call. This creates a child process that is an exact duplicate of the Redis parent process. At the moment of the fork, both the parent and the child point to the exact same memory addresses. The child process then begins slowly writing that memory to disk. But, what if the parent process receives a write command and needs to modify that memory while the child is still saving it? The kernel handles this using a mechanism called copy-on-write. If the parent tries to modify a page of memory, the OS intercepts it, creates a duplicate of just that specific page, and lets the parent modify the new one. The child process remains completely undisturbed, seeing the memory exactly as it was at the exact millisecond the fork occurred. The main thread never blocks, and the snapshot gets safely saved to disk. The second persistence method is AOF, or append-only file. While RDB takes snapshots every few minutes, AOF is a continuous log. Every time a write command is executed, like set key value, Redis appends that exact command to a file on disk. If the server crashes, Redis can perfectly reconstruct the data set by reading the AOF file from top to bottom and replaying every single command. In production, these two methods are often used together. RDB gives you fast restarts and compact backups, while AOF guarantees you won't lose the last few minutes of data if the power goes out. When you put all these pieces together, the in-memory architecture, the rich data structures, the lock-free single-threaded event loop, and the background persistence mechanisms, you realize why Redis is so ubiquitous. It's not just a cache, it's a highly optimized system for solving distributed architecture problems. If you need to enforce API rate limiting, use Redis string with an expiration timer. If you need to pass messages between microservices, use Redis pub/sub or Redis streams. And if you need to prevent users from booking the same seat simultaneously, use Redis atomic transaction. Redis forces you to think differently about how you store data. Instead of dumping everything into rigid relational tables and relying on slow disk reads, it gives you low-level data structures optimized for the speed of RAM wrapped in a network interface. So, the next time you type docker run Redis, you aren't just spinning up a temporary bucket for cache misses, you're spinning up one of the most elegantly engineered pieces of software in the modern stack. Let's say we have a web application which is used by 20 users daily. One day 10,000 users want to use your app at the same time. But your server can't handle that much traffic and starts responding with errors. The problem here is not in the hardware or network. It is caused by how software handles connections. But what exactly is a connection? When a browser wants something from the server, it first opens a channel. That open channel is the connection. The browser sends requests through it like for a JavaScript file and the server uses that same channel to send the data back. One connection can carry many requests one after another. So it isn't the same thing as a single request and it stays open the whole time even when nothing is being sent. In the late 90s and early 2000s, web servers used a thread per connection model. So when a user connects, the server spawns a new thread or process to handle them. that thread stays alive as long as the connection remains open. If you have 100 visitors, you have a 100 threads. If you have 10,000 visitors, you need 10,000 threads. Every thread requires its own stack memory. Even if you shrink the stack size aggressively, a thread might still take a megabyte of RAM. 10,000 threads mean 10 GB of memory consumed just to keep the connections open before the application even does any work. Another problem comes down to CPU limitations. A processor can only execute one thing at a time per core. To fake parallel execution, the operating system rapidly switches between threads, saving the state of one thread and loading the state of another takes time. At 10,000 threads, the CPU spends most of its cycles just switching contexts instead of serving web pages. This is a well-known problem with its own name, the C10K problem. The C is for concurrent, the 10K for 10,000 connections. A Russian engineer hit exactly this problem at Rambler. the search portal he worked for and he created engine X to fix it. This thread per connection model is wasteful because a connection spends most of its life waiting. It waits for the client to send a request, waits for the disk or a backend to answer, waits for the network to carry the bytes back. The whole time the thread tied to that connection is blocked. It can't pick up anyone else. It just holds its memory until that one connection finally needs something. Engine X uses a single process to watch thousands of connections at the same time. At any moment, most of them are idle. So, it only does anything when one of them has data coming in or a response going out. It handles that connection, then moves to the next one that needs something. The process never stops to wait on a connection the way the old thread did. So, one process keeps thousands of them going. When you start engine X, it creates one master process and multiple worker processes. The master process does very little actual work. It reads the configuration file, binds to the network ports, and spawns the workers. The workers do the actual job of handling network traffic. You usually configure Engine X to run one worker process per CPU core. If you have an 8 core server, you run eight workers. This specific ratio is intentional. When we have one worker per core, the operating system rarely needs a context switch between them. The worker process stays on the CPU and runs constantly to process network requests without interruption. We have one worker responsible for thousands of connections, but how does it know which ones actually need attention? most of them are doing absolutely nothing. So it can't waste time walking down the whole list and checking each connection one by one whether it has data. So instead of checking each connection itself, the worker lets the kernel do the watching. The kernel receives all the network traffic anyway. So it already knows which connections have data and which ones are idle. The worker uses a Linux facility called eole which keeps two lists that the worker and the kernel use to talk to each other. The first one is the interest list where the worker writes down which connections it wants the colonel to watch. The second one is the ready list where the kernel writes back which of those connections actually have something happening. The interest list gets built up as connections arrive. When a connection opens, the worker registers it once with a call named epolect and that adds it to the interest list. The kernel needs this list to know which connections belong to this worker. Meanwhile, the ready list is filled by the kernel. When data arrives for a connection that is on the interest list, the colonel adds that connection to the ready list. Let's look at a real example. A user opens the web app and that opens a connection. The worker sees the new connection and uses eole to tell the kernel to watch it. So the connection lands on the interest list. Then the user clicks the orders tab and the front end sends a request to the server to fetch those orders. That request travels over the connection that's already open. When the kernel spots the incoming bite, it checks the interest list and immediately flags the connection on the ready list. Once the connection is on the ready list, the colonel's job is done. The worker still has to read the bytes, fetch the orders, and send the response. To actually process the work, the worker runs an endless loop. The first thing it does is call eul wait. This asks the kernel for connections that are sitting on the ready list. If our connection is there, the worker grabs it, handles the request, and ships the data back to the user. Then the whole cycle starts all over again. If the ready list is empty, e-le blocks the endless loop. The worker falls asleep on that exact line of code without draining any processor resources. The loop doesn't just run over and over asking the kernel if anything changed. It asks once and waits for the kernel to wake it up when a new request arrives. This endless cycle runs for the entire life of the process. And this is called the event loop. Since the worker never wastes time on idle clients, Engine X can handle 10,000 connections at once. The event loop seems like a perfect solution. But it has one big problem. If the worker gets stuck on one connection, all other connections have to wait. Let's suppose your server stores large video files on a slow mechanical hard drive. One day a user requests one of those videos. So the connection arrives and the worker asks the drive for the data. Because the disc is slow, the worker sits there doing absolutely nothing, just waiting for the hardware to read the entire file and respond. That same worker is responsible for thousands of other active connections. While it waits for the video, everyone else is blocked until the disc responds. Someone else might be asking for a simple JavaScript file, but they won't get it until the slow disk read finally finishes. The system needed a way to handle slow operations without forcing everyone to wait. So in order to deal with those cases, developers built thread pools. When the worker receives a request that requires a slow disk read, it refuses to process it directly. It pushes the task to a queue and a separate background thread picks it up. The background thread takes over the job of talking to the slow disk, leaving the main worker free to return to the event loop. It goes right back to calling eole weight and serving the other users. Once the background thread finally pulls the file from the disk, it puts the data into memory and sends a signal. The next time the main worker checks the ready list, it sees that the data is ready. It grabs the bytes, ships them to the user, and immediately moves on to the next task. The main process never had to stop and wait for a slow operation that would block all other connections. I said earlier that the worker grabs the connection, handles the request, and ships the data back. But I think we need to define what handling the request actually means. So the last thing I want to cover in this video is what Engine X is actually responsible for. Engine X does not run your application code. It acts as a reverse proxy. When a request arrives, it opens a separate connection to your backend like a node API. It forwards the incoming data to node, waits for the backend to finish its work, and relays the response back to the user. Because engine X already sits in the middle and intercepts all the traffic, you can also use it as a load balancer. You just run multiple instances of your application and it will distribute the request across all your servers. So, the old approach of giving every single connection its own thread is what kills web servers the moment traffic spikes. Engine X completely changes this dynamic by relying on an event loop, allowing a single worker process to efficiently manage thousands of clients at the exact same time. Instead of wasting massive amounts of memory and processing power on idle connections that are doing absolutely nothing, it only uses server resources when there is actual data ready to be processed. 4. Git-explaination-teaching.txt
Git is one of those things we use every day, but have you ever wondered how it works under the hood? Most of us just memorize a couple of commands and hope nothing will break. Basically, Git is just a small key-value store plus a few pointers. Today, we are going to dive straight into that data model, and once you see it, every command will start to make sense. Everything Git knows about your project is stored in a hidden directory called .git. If you delete that folder, Git simply stops tracking your files, and your whole project history just disappears. If you look inside that directory, you will find four main things. An objects folder, a refs folder, a file called head, and a file called index. These four things create the entire system. As I mentioned at the beginning, Git is simply a key-value store. You hand it some content, and it gives you back a 40-character hash. That hash becomes a key you use to retrieve your data later. Because the key is generated directly from the data itself, we call this a content-addressable storage. We can prove it using plumbing commands, which low-level tools Git uses behind the scenes. If you pass a string to the Git hash object command, Git compresses it and saves it straight into the objects directory. It returns the hash, and if you give that hash to the Git cat-file command, you get back the exact same data you saved. Content addressing gives Git two main advantages. First, it stores identical content exactly once. If you have 10 identical files in different directories, they all produce the same hash, so Git only saves one copy. The second advantage is that Git catches data corruption instantly. If a single bit flips on your hard drive, the hash no longer matches the content, and Git knows right away. Git uses three main object types to build your repository. Technically, there's a fourth one called the tag object, but since it only puts labels on commits, we can skip it today. The first one is the blob, and it stores the raw contents of a file and nothing else. It has no file name, no file path, and no timestamps. It's just the raw bytes of your code. But this creates a problem. If blobs are completely anonymous, how does Git know which hash belongs to which file? Git solves this missing context with a second object type called a tree. A tree is Git's version of a directory. It's only job is to map file names back to those anonymous blob hashes. If you inspect a tree object, it reads like a simple list. Every line follows exactly the same pattern. It starts with the file mode, followed by the object type, which in this case is a blob. Next comes a hash, and at the very end of the line, you can see the actual file name. This is how Git finally connects a name to your raw data. But, repositories are not just flat lists of files. You usually have folders inside folders. Git handles this by letting a tree point to another tree. If there is a subdirectory in your project, the tree adds a line for it. The structure stays the same, but the object type changes from blob to tree, and the hash now points to that nested tree instead of a file. By linking trees to trees, Git builds the exact directory structure of your project. A single root tree describes the complete state of your project at a specific moment in time. It defines the exact hierarchy of files and folders. But, the tree tells us nothing about when this state existed, who created it, or what came before it. So, Git records history with the commit object. It is just a tiny text file, but if you inspect it, you see exactly what Git uses to build the graph. The first line starts with tree, followed by the root tree hash. The second line starts with parent, followed by the previous commit hash. After that, it records the author, committer, and a timestamp. Finally, at the end of the file, you get your actual commit message. The parent pointers chain these commit objects together into a directed acyclic graph. That graph is your project history. Every commit points backward to the commit that preceded it. Since every commit contains a hash of its parent, changing anything in the past changes every hash after it. This Merkle style structure is the same idea that blockchains are built on. You cannot modify an old commit without rewriting everything that comes after it because the new hashes cascade all the way down the graph. We usually assume a commit just stores what changed since the last one. We think of version control as a stack of patches, but it does not work that way. Every single commit points to a full snapshot of your entire project. When you run git diff or check a pull request on GitHub, you are looking at line-by-line changes. But Git does not store those diffs anywhere. It calculates them on the fly by grabbing two trees and comparing their hashes. Storing a full snapshot for every commit sounds extremely inefficient, but unchanged files produce the exact same hash. A new commit simply reuses almost everything from the previous one because the new root tree just points back to the existing blobs. To clean up the rest of the redundant data, Git runs garbage collection automatically whenever your repository grows past certain thresholds. It compresses loose objects into pack files and saves similar objects as deltas to save disk space. So, diffs actually do exist, but they are strictly a storage trick. They remain completely invisible to the core data model. Once you see how commits work, branches start to look completely different. We usually treat branches as heavy parallel timelines, but in Git, a branch is literally just a 41-byte text file sitting in the heads directory. It contains exactly one thing, a single commit hash followed by a newline character. That is the entire implementation of a branch. If branches are just text files, how does Git know which one is active? Every project has one file called head, which basically controls what you see in your working directory. It simply holds the path to your current branch, and that branch file points to a commit hash. When you create a new commit, Git saves a new commit object in the objects folder. Then, it reads head file to see which branch you are on. It opens that branch text file and replaces the old hash with your new commit hash. Basically, that's how a branch moves forward. Let's take a look at another example, which is merging. Say you merge a feature branch into main, and nobody has added any new commits to main while you were working. In that case, Git can do a fast-forward merge and it does not create a single new object. Git just opens the feature branch text file and copies the commit hash. Then it overwrites the hash inside your main branch text file. The pointer slides forward, Git checks that state out into your working directory, and the merge is complete. But what if someone did push new commits to main in the meantime? Now, the two histories have diverged. So, there is no straight line for the pointer to slide along, and a fast-forward is impossible. So, Git performs a real merge instead. It finds the common ancestor of both branches, combines the two snapshots, and creates a brand new merge commit. And that merge commit is special in exactly one way. It has two parent lines instead of one. Only then does Git move your main branch pointer to this new commit. Sometimes you check out a specific commit instead of a branch. In that case, Git updates head to point directly to that raw commit hash. That's called the detached head state. You're operating without a branch pointer, and if you make a new commit, Git will update the head file. Between your working directory and your permanent history sits the index. It is just a single binary file. This file acts as your staging area, holding the exact list of everything your next commit will contain. When you run Git add, Git immediately hashes your file. It creates a new blob and writes the raw bytes straight into the objects directory. Then it updates the index file to connect your file path with that new hash. Then when you run Git commit, Git reads the index and then converts your staged entries into tree objects and saves them straight into the objects directory. Then it wraps a new commit object around that root tree and links it back to your current commit. The last step is just a pointer update. Git reads head and moves your branch forward. Once you see the objects and the pointers, the commands finally start to make sense. Let's look at a common command like Git reset. You are simply moving the current branch pointer to another commit. Usually an older one. The flags you pass just dictate what happens to your actual files. Running get reset soft moves the pointer, but leaves your index and working directory alone. The mixed option moves the pointer and updates the index. And get reset hard moves the pointer, syncs the index, and completely wipes your uncommitted changes in tracked files. Untracked files survive even a hard reset. Another example is get rebase. The name implies you're moving commits around the graph, but get objects are immutable and nothing is ever moved. Get actually reads the changes from your old commits and creates brand new commit objects with entirely new hashes. Then it simply points your branch at the new commit. Your old commits remain completely untouched. They still sit exactly where they were in the objects directory. Objects in Git are immutable and append-only. Commands that feel destructive almost never actually delete data. They simply move pointers away from commits you no longer want to see. Git tracks all of these pointer movements. The reflog records every move that head makes in your local repository over the last 90 days, which is the default expiry. If you run get reset hard and lose a commit, or if you mess up a rebase, your original commit still exist. They are exactly where they were, just one reflog lookup away. Let's summarize everything that happened in this video. Git is just a content-addressed object store. Three simple object types and a few movable pointers.
